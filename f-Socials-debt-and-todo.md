## Claim-verification-router (precision evidence) — shipped

Replaces Stage 3's old "first provider with citations wins" `evidence.gather` loop with a staged precision pipeline wrapped around the **existing** provider chain, inside the existing Worker/Pipeline (no new microservice, no weighted scoring, no 8-variant query explosion). The governing metric is the **False_Evidence_Rate** — a wrong citation is worse than a missing one.

- [x] **Six-stage router** (`src/router/`): `normalize` (Claim_Type + Fact_Checkability triage), `queryPack` (4–6 purpose-distinct variants), `retrieve` (over the existing chain, tier set from the policy), `validate` (each candidate classified against the **original** claim, never a query variant), `outcome` (gates + routing + Evidence_Outcome), `audit`, and the `verifyClaim` orchestration.
- [x] **Three independent binary gates, no combined score** — Match_Type, Match_Confidence, and Source_Tier are recorded as distinct signals; only `same_claim` / `contradictory_but_relevant` candidates whose tier is **not** `excluded` enter the Claim_Ledger. `same_topic_different_claim` → Useful_Context; `background_context` → Context_Card; `irrelevant` discarded.
- [x] **Honest no-evidence.** `no_sufficient_evidence` (and `relevant_context_only`) carry zero citations and remain a valid served `ready` outcome; a single unmatched claim never fails the report.
- [x] **Invariant gate preserved + boot guard.** `core/assemble.ts` is unchanged; the router satisfies the gate by construction (strength ≠ `none` ⇔ ≥1 citation), and `assertInvariantGateIntact()` runs once at worker boot, refusing to start if the gate's pinned behavior ever weakens.
- [x] **Per-claim audit records persisted.** New `audit_records` table (migration `003`) + `Repository.saveAuditRecord` (memory + Postgres, best-effort so it never blocks a ready report). Each record is lossless: Original/Canonical claim, Claim_Type, fact-checkability, the Query_Pack, every raw candidate with Match_Type/confidence/Source_Tier/rank, and the final Evidence_Outcome.
- [x] **Neutrality enforced.** Static checks assert no Ad Fontes / AllSides / MBFC dataset is imported and that `VerifiedClaim`/`AuditRecord` expose no content-truth verdict or creator-reliability field.
- [x] **Tests.** All 16 correctness properties as `fast-check` PBTs (≥100 runs each, seeded deterministic normalizer/validator mocks so the router's logic — not the LLM — is under test), plus example/integration/neutrality tests. `fast-check` added as a dev dependency.
- [~] **Offline benchmark + Ship_Gate.** `src/router/benchmark/` computes `False_Evidence_Rate` per strategy (`current_chain` vs `router`, extraction model held constant) with a Ship_Gate (`FER_router ≤ FER_current`). The router is wired **live**; the current `fixtures.json` (~92 claims) uses synthetic `.example` URLs, so **running the benchmark on a real labeled set is the next step before relying on the gate decision in production** (folds into the educator-pilot labeled-claim collection).

## Report-graph normalization (dual-write) — shipped

Activates the reserved `claims`/`citations`/`perspective_links` tables so every persisted report also projects into normalized rows for cross-report analytics — *before* the first aggregate feature, as the roadmap (§7) committed. Additive and lazy: a pure projection module folded into the existing `Repository.saveReport` on both drivers (zero call-site changes), an ordered migration, a one-shot backfill, and the property/example/integration/smoke suite.

- [x] **Pure projection module** (`core/reportGraph.ts`). `projectReportGraph(report)` is a deterministic, I/O-free mapping from an `AnalysisReport` to its `ClaimRow`/`CitationRow`/`PerspectiveRow` set: one claim row per claim (`claimUid = Claim.id` for stable traceback), one citation row per citation linked by `claimUid`, one perspective row per perspective. A `none`/zero-citation claim projects a claim row with zero citation rows; non-normalized fields (`evidenceDescription`, `whyIncluded`) stay in JSONB only. Carries a guarded inline self-check.
- [x] **Dual-write folded into `saveReport`** (memory + Postgres). The JSONB payload stays the authoritative render source of truth, written and committed first exactly as before; the normalized rows are a separate, **best-effort** write. Postgres runs a single delete-then-insert transaction (idempotent replace, atomic per report); a failure is logged with the `report_id` via `console.error` and **never rethrown**, so the report stays served from JSONB. The memory driver mirrors it with public `claimRows`/`citationRows`/`perspectiveRows` accessors (like `disputes`/`flags`/`auditRecords`), keeping the offline path testable.
- [x] **Migration `004_report_graph.sql`.** Additive DDL after `003`: `claims.claim_uid` + a `(report_id, claim_uid)` unique index (the idempotent-replace natural key), plus cross-report query indexes on `citations(source_url)`, `citations(source_tier)`, `claims(claim_text)`, `perspective_links(source_tier)`. Preserves all existing `analysis_reports` data and JSONB consumers; no creator-reliability column.
- [x] **Backfill command** (`src/scripts/backfill.ts`, run via `tsx`). Iterates `listReportIds()`, skips reports that already have rows (`hasReportGraph`), and re-saves the rest through the same idempotent dual-write. Reads JSONB only, never mutates it; a per-report failure is recorded (`report_id`) and the loop continues; logs `{ processed, skipped, failed }`.
- [x] **Neutrality enforced.** Tiers attach to `citations`/`perspective_links` only — there is structurally no creator input to read. A static `Neutrality_Check` asserts the row types, projection source, and migration carry no creator-reliability dimension; a property test confirms no claim row carries a tier.
- [x] **Invariant gate untouched.** `core/assemble.ts` is verified, not edited — generators produce gate-valid reports and the projection faithfully reflects that state without re-deriving the gate. A guard smoke test asserts the gate is unmodified, migration ordering is correct, and `writeReportGraph` uses `$1..$n` placeholders only.
- [x] **Tests.** Eight `fast-check` PBTs (≥100 runs: dual-write, cardinality + linkage, round-trip field consistency, non-normalized retention, idempotent replace, backfill, source-only tiers, gate-faithful reflection) against `InMemoryRepository`, plus example tests (durability under normalized-write failure, backfill failure continuation, offline wiring), static smoke tests (neutrality, guards, interface parity), and 3 Postgres integration tests (migration non-destructiveness, transactional atomicity, cross-report `GROUP BY` queryability) that skip without a test DB. Offline suite 109/109 green; `npm run typecheck` clean.

## Parallel evidence lookups (bounded concurrency) — shipped

Removes the last serial bottleneck in Stage 3. The claim loop (`pipeline/stages.ts`) ran the router per claim in series, and the router (`router/index.ts`) submitted each Query_Variant to the chain sequentially — so the router migration's higher per-claim call count dominated cache-miss latency. Both loops now run in **bounded parallel** under one shared per-report counting semaphore, with no new dependency and the invariant gate untouched.

- [x] **One shared counting semaphore** (`src/concurrency.ts`, ~30 LOC). A FIFO-waiter counting semaphore is the `Bounded_Scheduler`: created once per report in `runPipeline`, threaded into every claim's `verifyClaim`, and acquired around each `retrieve(variant)` call. Because the pipeline wires the chain as a single retrieval source, **one gated retrieve = one provider submission**, so concurrent provider submissions are `≤ cap` by construction regardless of how work splits across the two loops. `run()` releases its slot in `finally` (frees even on throw, so a failing lookup never deadlocks or starves siblings). Chosen over nested per-level caps (which bound the global count only as `C_claim × C_variant`) and over adding `p-limit`/`p-map`.
- [x] **Configurable cap** (`src/config.ts`). Pure `resolveConcurrencyCap` + `config.concurrencyCap`: valid = an integer in `[1,32]`; absent/empty/non-numeric/non-integer/out-of-range falls back to the documented default **4** (rejected to default, **not** clamped) and `console.warn`s naming `CONCURRENCY_CAP` (warn-and-default, never aborts startup). API and worker read the identical value from the single config module; `worker.ts` passes it into `runPipeline`.
- [x] **Determinism by construction.** Lookups are *scheduled* in parallel but results are *written into pre-sized arrays at their original index* (claims by extraction index, candidates by Query_Variant position) and flattened in index order — completion order never touches output order. `retrievalRank` stays assigned from in-variant position inside `makeRetrieve`. `cap=1` reproduces the exact serial baseline.
- [x] **Error isolation reproduces serial defaults exactly.** A variant whose retrieve throws contributes zero candidates (`perVariant[j] = []`); a claim whose every variant fails resolves to `no_sufficient_evidence` with zero citations; a failure in one claim/variant never shifts, drops, or reorders the rest and triggers **no** extra provider call or retry. `normalize`/`triage` run before and outside the semaphore, so a `not_fact_checkable` claim short-circuits with zero acquisitions.
- [x] **Invariant gate + `makeRetrieve` are verify-only.** `core/assemble.ts` and `makeRetrieve`/`DEFAULT_PER_VARIANT_CAP` are unchanged; the existing `assertInvariantGateIntact()` worker boot guard stays in place. The parallel path changes *when* claims/citations are produced, never *which*.
- [x] **p95 ship-gate reporting.** Added a pure nearest-rank `percentile`, a `LatencyBenchmarkReport` (`latenciesMs`, `p95Ms`, `thresholdMs`, `passed`), and `runLatencyBenchmark(runOnce, { runs ≥ 20, thresholdMs = 30_000, now })` to `router/benchmark/runner.ts` (injected clock → deterministically testable; FER machinery untouched). The actual ≥20-run cache-miss benchmark execution against the fixture set is an ops/integration step; this delivers the `passed` verdict logic.
- [x] **Tests.** Eight `fast-check` PBTs (≥100 runs each, offline mock substrate, `node:test` + `node:assert`): Property 1 output-equivalence (`deepEqual(report(cap=1), report(cap=N))`, ids excluded), 2 error isolation under injected failures, 3 in-flight ≤ cap, 4 cost-neutrality (call-count cap-invariant incl. `not_fact_checkable` and failures), 5 honest-none preserved, 6 latency reduction under bounded parallelism, 7 p95 reporting, 8 cap resolution. Plus runnable assert-based self-checks in `concurrency.ts` and the benchmark runner. No new dependency (`fast-check` already a devDependency). Offline suite **120/120** green; `npm run typecheck` clean. *(Hardening, expert-review-queue session: Property 6's latency check was rewritten from a wall-clock comparison — which could flake under heavy machine load when the two sequential runs were scheduled unevenly — to a deterministic logical "wave" clock. Each mock lookup now costs one `setImmediate` wave instead of a real `setTimeout`, so cap=1 yields one lookup per wave (= total lookups) and cap=N yields `ceil(total/N)`, making `parallel < serial` true by construction and load-independent; verified 15/15 green under 4 concurrent CPU-bound jobs. Also faster — no real sleeps.)*

## Observability instrumentation (Sentry + PostHog) — shipped

Wires error/exception monitoring (Sentry) and product/event analytics (PostHog) into the two server processes and the web app, so unhandled errors and pipeline failures are captured with debugging context (Stage, report id, provider category) and the two **red-line trust KPIs** — **Citation_Coverage** and **Model_Human_Agreement** — become measurable before pilots (roadmap §7's prerequisite). An **additive, observe-and-verify-only** layer: it never alters analysis behavior, never edits `core/assemble.ts`, and the offline/zero-key path runs exactly as before. New deps: `@sentry/node`, `posthog-node` (server), `posthog-js` (web).

- [x] **`Telemetry` port + composition-root selection.** A new `Telemetry` interface in `src/infra/ports.ts` (two **synchronous `void`** methods, `emit`/`capture` — fire-and-forget, never awaitable) alongside `Cache`/`Queue`/`Repository`/`RateLimiter`, selected by `selectTelemetry()` in `compose.ts` exactly like the other infra drivers and exposed on `AppContext`. Both `index.ts` and `worker.ts` obtain it only through `buildContext()`, so the two processes never drift. Config (`config.sentryDsn`, `config.posthogKey`) is read once in `config.ts` (verbatim-trimmed, `''` when unset/whitespace-only) and is **never** added to `missingRequiredConfig` — absent telemetry config can never block startup.
- [x] **Offline-first / fail-open by construction.** Zero keys ⇒ a frozen `No_Op_Telemetry` singleton (`infra/telemetry/noop.ts`) that accepts every call, emits nothing, opens zero connections. Each backend activates independently (exactly-one-configured activates only that one). Every active send is wrapped in a `safe()` helper that swallows any synchronous throw / async rejection (logs once) and never awaits — a telemetry fault can never reach the pipeline, the HTTP path, or startup. An init failure warns and degrades to no-op rather than aborting.
- [x] **Vendor SDKs isolated to one module.** `@sentry/node` / `posthog-node` are imported **only** in `infra/telemetry/active.ts`; every other server module speaks the `Telemetry` interface. A boot-smoke check statically asserts no other `src/**` file imports a vendor SDK.
- [x] **PII / secret redaction at the boundary.** A pure, total `Redactor` (`infra/telemetry/redact.ts`) deep-copies every payload and strips Denied_Fields (transcript, raw claim text, JWTs, API keys, user identifiers — case-insensitive key match + value scrub) at any depth, cycle-safe via a `WeakSet`. `emit` runs payloads through the Redactor **and** the Neutrality_Guard; `capture` through the Redactor only (an error context is not a product event). A residual-denied payload is suppressed with a redaction-failure indication.
- [x] **Neutrality enforced (lens, not a judge).** A pure, total `Neutrality_Guard` (`infra/telemetry/neutrality.ts`) withholds any event carrying a creator/author/person/channel reliability dimension or a content-truth verdict; a source tier passes **only** when it describes a source/citation and is not co-located with a person identity. Disputes/flags are referenced by report+claim id only — never a user identity, never attached to a creator/channel.
- [x] **The two red-line KPIs derivable.** Pure, total, deterministic functions in `core/kpi.ts`: `citationCoverage(audits)` = cited-outcome claims / claims carrying one of the six defined outcomes (∈ `[0,1]`; empty ⇒ `0`; out-of-enum excluded; honest-none counts toward the denominator only), and `modelHumanAgreement(outcomes, signals)` pairing model outcomes against expert-review/flag/dispute signals by (reportId, claimId) (`undefined` for lack of signal, else agreeing/compared ∈ `[0,1]`). Neither recomputes or alters an Evidence_Outcome.
- [x] **Call sites (additive only).** Worker `handleJob`: `capture(err, {reportId, stage, providerCategory})` exactly once on failure (the `failed` persist unchanged), and one `pipeline_complete` event (status, `durationMs`, outcome distribution, citation coverage) emitted **after** `runPipeline` returns — never on the gate path. Express error handler: `capture` once, HTTP status/body unchanged. Routes: `cache_hit`/`cache_miss`/`dispute`/`flag` events carrying ids + categorical labels only (no user id).
- [x] **Web analytics (consent-gated).** `apps/web/src/analytics.ts` — a tiny `posthog-js` wrapper: a pure `buildWebEvent` (web analogue of Redactor + Neutrality_Guard), `track`/`grantConsent`/`hasConsent`. No `VITE_POSTHOG_KEY` ⇒ synchronous no-op, zero outbound. Emits nothing until consent is granted; `track` has no render side effects so the DOM/routing/a11y tree are identical active vs no-op. Wired into the App hash-route view effect and the dispute/flag/share handlers.
- [x] **Invariant gate untouched.** `core/assemble.ts` is verified, not edited; telemetry is emitted only after the pipeline produces a report, so the gate path makes zero telemetry calls by construction. The existing `assertInvariantGateIntact()` boot guard still runs.
- [x] **Tests.** Thirteen `fast-check` properties (≥100 runs each, offline, zero outbound) — redaction no-denied/idempotence/preservation/totality, neutrality correctness+totality, no-op zero-emission, active fail-open, pipeline telemetry-invariance (active vs no-op report is field-by-field equal), citation-coverage and model-human-agreement bounds, unknown-context substitution, config-never-blocks-startup, and the web no-PII/no-op property — plus example tests (selection branches, port wiring, call sites, web consent/interaction) and boot-smoke extensions (no-op-before-serving, single vendor-SDK touchpoint, gate intact). Server offline suite **164/164** green, `tsc --noEmit` clean; web Vitest **35** green, `tsc -b` clean. One real fail-open gap (an `emit` path that didn't return the PostHog promise, leaking a rejection) was caught by the Property 7 test and fixed.

## Expert review queue (review workflow) — shipped

Builds the review **workflow** on top of the existing dispute/flag **intake** (which intentionally lagged it). Authorized reviewers can now list incoming disputes and flags as one queue, claim/release items to avoid duplicated work, and record a bounded, neutrality-safe resolution — which feeds the report's existing `Provenance.reviewStatus` (`ai-generated` | `expert-reviewed` | `under-dispute`). Additive and offline-first: review state rides as columns on the existing `disputes`/`flags` tables, reached only through new `Repository` methods; the report is **never written** — its review status is **derived on read** — so the invariant gate is preserved by construction.

- [x] **Additive schema (migration `005_review_workflow.sql`).** Two enum types (`review_status_kind`, `resolution_outcome`) behind re-run-safe `DO/EXCEPTION` guards, plus six additive `IF NOT EXISTS` columns on **both** `disputes` and `flags` (`review_status NOT NULL DEFAULT 'pending'`, `assigned_reviewer`, and the resolution quartet `resolution_outcome`/`resolution_note`/`resolved_by`/`review_resolved_at`) + review-status/report indexes. Legacy `status`/`resolution`/`resolved_at` untouched; existing rows default to `pending` with no backfill; no creator-reliability or truthfulness column.
- [x] **Four `Repository` methods, both drivers** (`ports.ts` + memory + Postgres). `listReviewItems(filter?)`, `claimReviewItem(id, reviewer)`, `releaseReviewItem(id, reviewer)`, `recordReviewResolution(id, resolution)` returning a discriminated `ReviewActionResult` (`ok` | `not_found`/`conflict`/`not_actionable`). Review_Item id is `"{kind}:{sourceId}"`. Claim is an **atomic compare-and-set** (Postgres conditional `UPDATE`, in-memory no-`await`-between-read-and-write) so exactly one of N concurrent reviewers wins and a same-reviewer re-claim is idempotent; resolution requires no prior claim and overwrites in place (never duplicates). Postgres is **parameterized SQL only** (`UNION ALL` queue projection, `$n`-bound everywhere; table name from a hardcoded kind whitelist). The memory driver exposes review state via the existing public `disputes`/`flags` accessors, keeping the offline path testable; no submitter identity is ever projected onto a Review_Item.
- [x] **Derived-on-read report status, gate preserved.** A pure `deriveReportReviewStatus(current, itemStatuses)` (`core/reportReviewStatus.ts`): empty → unchanged; any non-`resolved` → `under-dispute`; all `resolved` → `expert-reviewed`; never a new status value. The read path (`GET /analyses/:id`, `GET /r/:slug`) overlays the derived value onto the **outgoing response only** — the persisted report is never rewritten and no gate-relevant field is read or touched, so `core/assemble.ts` is verified, not edited (Property 8 proves the gate verdict is unchanged under the overlay).
- [x] **Role-gated routes + fail-closed auth.** Four routes under `/api/v1/review` (`queue`, `items/:id/claim`, `items/:id/release`, `items/:id/resolution`), each behind `requireAuth + reviewerGuard`. The guard reads `config.reviewerRole` (`REVIEWER_ROLE`, default `''`): empty ⇒ **fail closed**, every review route denied `403 reviewer_role_not_configured` with a degrade-and-warn startup line (never aborts startup); authenticated non-reviewer ⇒ `403 not_a_reviewer`. zod validation at the boundary (`reviewQueueQuerySchema`, `reviewResolutionSchema`); malformed `:id` → `400`; repo outcomes map to `404`/`409`.
- [x] **Neutrality enforced (lens, not a judge).** The `RESOLUTION_OUTCOMES` vocabulary (`core/reviewOutcome.ts`) is seven **framing/evidence-only** values (e.g. `framing_example_confirmed`, `evidence_overstated`, `context_gap_noted`) — no truthfulness verdict, no creator-reliability rating. A static `Neutrality_Check` scans the migration DDL, the outcome vocab, and the Reviewer Console labels for banned creator/verdict tokens (it caught and forced the removal of the word "verdict" from a console screen-reader string), and a property test pins the schema's accept/reject boundary.
- [x] **Web Reviewer Console** (`apps/web`, `#/review`). Lists items with report context, dispute reason / flagged technique, status, and an explicit "unassigned" label; keyboard-operable claim/release/resolve with ARIA descriptions; color-never-alone; ≤768px single column; distinct loading / empty-queue / error+sign-in states; honest text-labeled "no external review found" state with no substitute verdict. New `api/client.ts` functions mirror the `submitDispute`/`submitFlag` fetch+throw style; `#/review` wired into `App.tsx` (hash routing only).
- [x] **Tests.** Nine `fast-check` PBTs (≥100 runs each: queue projection fidelity, list filter+ordering, claim idempotence, claim exclusivity under contention, claim/release round-trip, resolution→item+report status, total/bounded status derivation, invariant-gate preservation under overlay, neutrality) plus the static neutrality scan, the authz/fail-closed/validation/edge/offline example suites, and the web Console + honest-absence tests. The Postgres migration test (non-destructive, re-run-safe, defaults existing rows to `pending`) lives in `test:integration`. Server offline suite **208/208** green, `tsc --noEmit` clean; web Vitest **47/47**, `tsc -b` clean.

## Progressive disclosure report UI — shipped

Closes the largest open block in the roadmap (§8 "UI/UX recommendations", with the light §6 borrowings). A **presentation-layer-only** change to `app/apps/web/src/components/Report.tsx`: it re-presents data the report already carried for progressive disclosure and reduced cognitive load. No report-schema change, no API-client change, and the invariant gate is read (`report.status`) but never invoked or edited. The lens-not-judge compass holds by construction — no verdict on content, no reliability rating on a creator; source-tier chips attach to sources/citations only.

- [x] **Summary lead + independent disclosure drawers.** The four mutually-exclusive tabs are replaced by a `SummaryLead` (rendered expanded on first paint — the TLDR plus the single most-important framing signal, marked with a soft amber underline **and** an adjacent text label so emphasis never relies on color alone) followed by five independent, keyboard-operable `DisclosureSection` drawers (Claim Ledger, Framing Signals, Useful Context, Other Angles, Issue-Frame Position), each collapsed on first paint and each owning its own open state (toggling one never affects another — the old shared `tab` enum is gone). Honest-absence: when there is neither a non-whitespace TLDR nor any framing signal, the lead states "No summary available for this analysis." and every other section stays collapsed.
- [x] **Mockup rationale blocks ported onto the live wiring.** A `RationaleBlock` renders the perspective `whyIncluded` ("Why included") and the claim rationale ("Why this is here") **verbatim** from fields already in the report — evidence description first, source basis as fallback (pure `claimRationale` helper), omitted entirely (no placeholder) when both are absent/whitespace-only.
- [x] **Three lens-aligned borrowings, all descriptive (never evaluative).** Ground News blindspot → `CoverageAngleNote` ("covered from one angle", naming the pole of each issue-frame axis with magnitude > 0.8, with a seek-other-perspectives directive only when perspectives exist; omitted when no axis is strong or no issue frame). AllSides → spatial `IssueFrameChip` (report-level + per perspective; truncates to 120 chars with the full label on hover/focus; never a verdict/rating/ranking/score). NewsGuard → `SourceTierChip` on citations and perspective sources rendering the human-readable `TIER` label (never the internal id, never on a creator).

## Supabase user sync — shipped

Closes the one legacy foreign-key seam left after accounts/save and workspaces moved to the subject-as-`TEXT` convention: the authenticated community **flag** still keys on `flags.user_id UUID NOT NULL REFERENCES users(id)`, and the flag route persisted the Supabase subject straight into it — which the Postgres driver rejects (no `users` row for that subject) and the FK-less in-memory driver silently accepted. **User_Sync** ensures a local `users` row keyed to the JWT subject exists, derived **solely** from the already-verified token claims (no Supabase round-trip), invoked at the flag trust boundary after payload validation and before the FK-bearing insert. Strictly additive and confined to three layers behind existing seams; the invariant gate in `core/assemble.ts` is never read or written, and no synced field expresses a content verdict or creator rating.

- [x] **Repository seam, both drivers** (`infra/ports.ts` + `memory.ts` + `postgres.ts`). New `ensureLocalUser({ id, email?, role? })` idempotent upsert + `getLocalUser(id)` read, with a lens-safe `LocalUser` projection (`id`/`email`/`role`/`createdAt` only — no verdict, no creator rating, by construction). At most one row per subject; on repeat the row's `id` and original `created_at` are preserved and `email`/`role` are merged from the claims, **retaining** a prior value when the claim omits it. Postgres is a single parameterized `INSERT … ON CONFLICT (id) DO UPDATE` (`created_at` never in the `SET` list; `COALESCE` retains stored values); the in-memory driver mirrors it with a private `Map` keyed by subject, atomic by construction on the single-threaded loop so concurrent syncs converge on one row. User_Sync SQL lives only in `postgres.ts` (no ad-hoc query elsewhere).
- [x] **Additive migration `009_user_sync.sql`.** The narrowest possible change — `ALTER TABLE users ALTER COLUMN email DROP NOT NULL` — so an email-absent synced user is storable with `email` NULL. Data-preserving (every existing row keeps its email), re-runnable (`DROP NOT NULL` is idempotent), and `UNIQUE(email)` is left intact (Postgres treats NULLs as distinct, so multiple email-absent users never collide). Sorts lexically after `008`. *(Flagged in the design: Req 9.1's "remove no constraint" is read as "data-preserving + re-runnable" here, since a synthetic sentinel email would violate the stored-as-null requirement and pollute the unique index; the only zero-`ALTER` alternative is re-keying the flag FK onto the subject as `TEXT`.)*
- [x] **Trust-boundary validation + flag-route wiring** (`http/validation.ts` + `http/routes.ts`). New `syncedIdentitySchema` (zod: `id` a non-empty UUID; present `email` a valid address ≤320 chars; present `role` non-empty ≤255 chars). The flag handler validates the verified identity → `ensureLocalUser` in a try/catch → `createFlag` with `userId = subject`. A malformed identity/claim → `400 invalid_identity` with **no** sync and **no** persist; a sync failure → `500 sync_failed` (generic body, telemetry capture with report id + stage only) with **no** flag persisted. The existing `201 { ok: true }` success body and `ON CONFLICT … DO NOTHING` flag idempotency are unchanged. Dispute and saved-report/workspace routes are untouched — they never call `ensureLocalUser` (subject-keyed flows stay subject-keyed; disputes stay anonymous with `raised_by` NULL).
- [x] **Scope boundary + offline-first preserved.** Save, workspace-create, and dispute-intake create no `users` row and invoke no sync; with zero keys `buildContext()` composes the in-memory Repository and a flag submission succeeds end-to-end with no network seam in `ensureLocalUser`.
- [x] **Invariant gate + compass intact.** `core/assemble.ts` is verified, not edited; a property test confirms the readiness classification is identical across an interleaved User_Sync, and the `LocalUser` shape carries no truthfulness-verdict or creator-reliability field (static + round-trip checks).
- [x] **Tests.** Ten `fast-check` properties (≥100 runs each: idempotent identity, claim-reflection-with-retain, concurrent convergence, email-absent validity/non-collision, flag-resolves-to-synced-user, flag idempotency preserved, malformed-identity rejection, sync-failure isolation, driver parity, readiness invariance) plus static guards (driver presence, SQL safety, neutrality) and example tests (offline wiring, scope isolation). Postgres-dependent **driver parity** and **migration** tests run under `test:integration` and were verified green against a real Neon Postgres (throwaway project); their migration-replay helpers tolerate the concurrent-creation race (a `23505` unique-violation on a `pg_*` catalog) alongside the usual "already exists". Server offline suite **353/353** green, `tsc --noEmit` clean.
