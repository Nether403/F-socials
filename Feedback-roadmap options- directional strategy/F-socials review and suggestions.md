# F-Socials Codebase Overview

The F-Socials repository is a TypeScript-based full-stack project. Its core is a Node/Express backend (in `apps/server/src`) that implements an analysis pipeline – converting input (URL or transcript) through stages (transcript ➞ extract ➞ evidence ➞ perspectives ➞ invariant *“lens-not-judge”* gate). The code is organized into modular providers and infrastructure layers: e.g. abstract `TranscriptProvider`, `EvidenceProvider`, etc., with default *mock* implementations and clear swap points to real services (e.g. Google Fact Check API, NewsAPI/GDELT, YouTube). The backend uses modern tooling – TypeScript with strict settings, Express for routing, Zod for input validation, BullMQ + ioredis for queuing/caching, and pg for Postgres (though defaults use in-memory storage). An (unused) React front-end lives in `apps/web`, with Vite, React 19 and lucide-react icons configured. 

The README notes that the service currently runs **offline** (no real APIs) and uses a hash-based cache to avoid redundant work. Future “slices” include authentication (via Supabase), rate limiting, a public dispute/review queue, and a shareable report URL. Importantly, the design vision is “**A lens, not a judge**”: the UI/UX style guide emphasizes *neutral*, evidence-driven feedback (no binary green/red truth verdicts) and encourages exploration via collapsible evidence panels and alternative viewpoints. This ethos is intended to combat sensationalism and cognitive bias by letting users inspect rather than be told what to think.

## Current Features & Architecture

The existing feature set is essentially an API endpoint for submitting content to analyze. Users (or the frontend) send JSON with either a `transcript` string or a `url` (for YouTube or article), and the server queues a job. The pipeline produces an **AnalysisReport** summarizing claims, supporting evidence, and related perspectives. The backend code (in `pipeline/stages.ts` etc.) orchestrates each step and finally applies an “invariant gate” to catch extreme or irrelevant content. The analysis result is stored and can be fetched via GET requests (e.g. `GET /api/v1/analyses/{id}` or a share slug route `/r/{slug}`). The repository includes basic tests (e.g. `test/invariant.test.ts`) to enforce non-negotiable logic checks.

No polished user interface is yet in place, but the scaffolded React app hints at one. The client code includes a `detectInput` function to auto-classify pasted text as a URL vs free transcript, and methods to submit an analysis and poll for completion. This suggests the intended flow: user pastes a claim or link, F-Socials runs the analysis, then displays a report. The design docs enumerate user journey maps and wireframes (in `Design concepts/`) covering onboarding, content discovery, and dispute submission. 

## Similar Fact-Checking and Anti-Polarization Platforms

Several existing tools offer related functionality. Key categories include balanced-news aggregators, fact-check databases, and browser plugins for credibility. Notable examples:

- **Balanced News Aggregators:** *AllSides* and *Ground News* curate multiple perspectives on current events. AllSides tags news stories by bias (Left/Center/Right) and shows the same topic from different viewpoints. Ground News similarly aggregates headlines but adds features like a “Blindspot” feed highlighting stories **over-reported by one side** (e.g. a story covered 100% by right-leaning sources in our example). These sites present bias breakdowns and counts for each source, giving users context on polarization.  
- **Fact-Checking Websites:** Traditional fact-checkers (PolitiFact, Snopes, AP Fact Check, etc.) publish vetted analyses of viral claims. For example, PolitiFact’s *Truth-O-Meter* rates claims on a true-to-false scale, while Snopes provides research articles with verdict icons. These are single-purpose sites, but F-Socials could integrate their data or mimic their detail (e.g. showing fact-checker logos or linking to PolitiFact articles when relevant).  
- **Crowdsourced Checkers:** *Our.News* is a crowd-checked system where users rate news items on trust, bias and sources; over time it builds a database of rated content. *Truth Goggles* (Stanford) is an experiment where participants read claims and then expert fact-check analyses, which measurably reduced partisan bias. These show the value of involving users in evaluation and educating them via transparent evidence.  
- **Browser Extensions and Plugins:** A variety of browser tools flag misinformation or bias in real time. For example, **Stopaganda Plus** tags articles with MediaBiasFactCheck’s reliability scores and bias labels. The **Trusted News** extension (Eyeo/Adblock) looks for language cues (“in our opinion”) to warn of biased phrasing. The **Factual** extension computes an algorithmic trust score for any article feed. **Logically’s** extension combines AI and expert review to rate article credibility, showing context about authors and sites. Many of these work via content scripts on social media or news sites. F-Socials could adopt a similar approach (e.g. providing an extension that displays its analysis for content shared on Twitter, Reddit, etc.).  
- **Media Source Ratings:** *Media Bias/Fact Check (MBFC)* rates news outlets’ overall bias and accuracy. *NewsGuard* employs journalists to rate site reliability on nine criteria. F-Socials could incorporate these ratings into reports (e.g. showing a NewsGuard score for each source it finds), helping users gauge trustworthiness.  
- **Specialized Tools:** A few research projects target polarization directly. For instance, a University of Washington study built a browser extension that **downgrades extreme partisan posts** in a social feed; users exposed to it reported feeling more warmth toward the other party. This kind of algorithmic de-polarization suggests adding sentiment or extremity checks. The **InVID** plugin provides verification tools for images/videos (metadata, reverse image search) – F-Socials could similarly verify media sources. “Who Targets Me” gathers political ad targeting info to expose echo chambers. These show the breadth of anti-polarization ideas available.

## Recommendations

**1. Strengthen and Modularize Architecture:** The existing pipeline is well-structured, but could be enhanced for scalability and real data integration.  
- **Real Providers:** Implement the planned swaps (Google Fact Check API, NewsAPI/GDELT, YouTube transcript API, etc.) to replace mocks. This will dramatically increase real-world utility. Use environment configuration (`CACHE_DRIVER`, `QUEUE_DRIVER`, `REPO_DRIVER`) to seamlessly toggle between in-memory and production services as planned.  
- **Microservices and Cloud:** Consider breaking the pipeline into microservices or serverless functions (e.g. a separate service for transcript processing, one for evidence search, etc.) to allow parallel scaling. Hosting the API on a robust platform (AWS/GCP/Azure) with managed Redis and PostgreSQL would handle high load.  
- **Data Storage:** Move from in-memory to persistent storage early. Use Postgres (as planned) for reports and queue state, ensuring reports survive restarts. Add a schema migration workflow (the repo includes an init SQL) to safely evolve the database.  
- **Observability & Testing:** Add logging/metrics (e.g. request rates, processing times) to monitor performance. Expand automated tests beyond the invariant gate – e.g. end-to-end tests of each pipeline stage. This will catch edge cases and keep the system reliable.  

**2. Refine Code Structure:** Ensure maintainability and clarity.  
- **Type Safety:** The project already uses strict TypeScript; continue this rigor across new modules. For example, define Zod schemas for all API inputs/outputs.  
- **Authentication & Security:** Implement token-based auth (supabase.js is stubbed in `src/auth/supabase.ts`). Protect endpoints with JWT middleware (JOSE library is already included) so only authorized apps/users can submit analyses or view reports. Rate-limit the `/analyses` endpoint (using BullMQ rate limiting or middleware) to prevent abuse.  
- **API Design:** Consolidate API versioning (v1) and consider a RESTful design: e.g. allow GET by `slug`, accept both JSON and form submissions. Provide clear error messages and status codes (the client already handles 429s and 404s).  
- **Documentation:** Add a clear OpenAPI/Swagger specification for the API. This helps frontend developers and potential integrators understand the contract. Update the README with usage examples (the existing curl example in [60] is helpful; maybe also show a JavaScript example using the client library).

**3. Enhance UI/UX Layout:** Make the interface intuitive and inclusive. Drawing from both the design vision and successful analogues:  
- **Multi-Perspective Display:** Like AllSides, present each claim’s analysis with perspectives side by side. For instance, if a claim has a contentious paraphrase, show how left-leaning, centrist, and right-leaning sources cover it. Use colored labels or icons (e.g. “Left/Center/Right”) as AllSides does.  
- **Evidence & Context Panels:** Follow the “evidence drawers” idea. For each extracted fact, allow expansion to reveal citations, links, or underlying data (e.g. quote from a fact-checker). This is akin to PolitiFact showing its research in detail. Offer a “Useful Context” section (the design doc mentions this) to give additional credible angles, similar to AllSides’ “perspectives” blog posts.  
- **Bias Indicators:** Without using aggressive red/green “true/false” coloring, still signal caution. The design doc suggests teal for confirmatory info and amber/coral for caution. Use these consistently (e.g. a yellow ⚠️ icon for unverified claims, green ✅ for well-sourced facts). Avoid binary verdicts; instead display neutral language like “Supported” / “Unsupported” / “Incomplete” with quantitative confidence.  
- **User Flow:** Streamline the input process as in the client code: allow pasting a link or text and auto-detect type. After submission, show a progress indicator (with polling) until results are ready. Then display the report. Provide “Share” functionality generating a short slug/URL (the code has a `/r/{slug}` route) so users can share analyses.  
- **Mobile-Friendly & Accessibility:** Ensure the layout is responsive (WCAG guidelines). Large fonts and sufficient contrast (the design notes call for a structured, scholarly typographic hierarchy). Support keyboard navigation and screen readers. Include multilingual support if aiming for diverse audiences (e.g. translations of UI text).  

**4. Feature Enrichment (Incorporate Best Practices):** Borrow effective elements from other tools. Some possibilities:  
- **Source Credibility Scores:** Integrate NewsGuard or MediaBiasFactCheck data to display a reliability score or bias rating for each source in a report. A small badge next to each cited link can quickly signal trustworthiness.  
- **Crowdsourced Feedback:** Allow users to flag errors or submit their own evidence (like Our.News or Logically). For example, implement a “Submit your evidence” button that adds to the knowledge base, possibly with a moderation queue (F-Socials already plans an “expert review queue”). This engages the community and improves coverage over time.  
- **Personalization vs Exposure:** Consider optional personalization controls. For example, like the UW study, give users a slider to control how much alternative-leaning content they see. That research found that downranking extreme content reduced partisanship. F-Socials could experiment with a “balance level” setting that, say, occasionally shows a contrary viewpoint even if it’s not immediately obvious.  
- **Gamification/Education:** Add an interactive “quiz” or trivia mode (like “Fake News: The Game”) to educate users about common misinformation patterns. This can reinforce media literacy, turning passive reading into active learning.  
- **Social Features (Carefully):** If privacy allows, consider optional user profiles where one can bookmark interesting reports or follow topics. However, given the mission, avoid any social-network style features that could inadvertently create echo chambers. The focus should remain on facts and context, not popularity contests.  

**5. Emphasize Inclusive, Non-judgmental Tone:** As the motto “lens-not-judge” implies, F-Socials should use neutral, respectful language. Summaries and labels in the UI should avoid moralizing. For instance, rather than calling a claim “totally false”, say “Evidence does not support this claim.” This aligns with the design principle of “calm precision”. Also, the project should ensure content is accessible to non-technical users – avoid heavy jargon, explain any technical term (e.g. “This claim has 3 sources on the left, 2 on the right”), and provide a glossary if needed.

In summary, F-Socials already has a solid backbone. By swapping in real data sources, refining the tech stack for production, and enriching the UI with multi-perspective and community-driven features (inspired by AllSides, Ground News, Our.News, etc. and by browser tools like Stopaganda and NewsGuard), the platform can evolve into a comprehensive, user-friendly fact-checking lens that counters polarization while remaining engaging and inclusive. Each improvement should be guided by user testing and accessibility standards to ensure the final product is as effective and unbiased as possible.
